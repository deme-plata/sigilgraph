Follow-up to the v0 announcement (Flux / SigilGraph / SigilOS, 2026-05-29).
Since then the cryptography-relevant claims moved from "wire shape locked +
crate-level tests" to "running against a live testnet, verifiable by anyone
with curl and a browser." Five items worth the list's attention.

== 1. The light client is real and running (verify whole-chain state in microseconds) ==

The v0 post described the tip-proof wire format. It now runs end to end against
the live SigilGraph testnet (network id sigil-g0, currently ~1.43M blocks).

A light client does NOT download blocks. It fetches a published tip:
{height, hash, roots:{wallet, dex, event_log, contract}} and INDEPENDENTLY
recomputes the canonical tip-proof fingerprint:

    fp = BLAKE3( version(u16 LE) || len(network_id)(u32 LE) || network_id
                 || height(u64 LE) || wallet_root || dex_root
                 || event_log_root || contract_root )      [each root 32 B]

This is byte-identical to what the producer's sigil-tip-proof crate computes.
Two independent implementations now agree on it on the wire:

  - a Rust light client (sigil-top, a terminal node monitor) reconstructs the
    canonical bytes and verifies in ~3 microseconds, 0 blocks downloaded, and
    follows the live network tip to lag 0 (100% "synced" in the lightweight
    sense: the verified height tracks the producer's tip, each tip checked).
  - a browser light client recomputes the SAME fingerprint with a vendored
    @noble/hashes BLAKE3 and shows the verdict; I checked the browser's
    fingerprint against an independent computation and they match exactly.

Cryptography-list-relevant correction from v0: the earlier browser node used
SHA-256 as a stand-in "integrity" hash. That was wrong for a BLAKE3 chain --
it hashed the HTTP body, not the canonical tip bytes, and used the wrong
primitive. It now computes the real BLAKE3 tip-proof fingerprint over the
canonical encoding. Honest scope unchanged: the Blake3Fingerprint flavor is
integrity + network-binding + non-corruption, NOT adversary-resistant on its
own (any peer can fabricate one). Adversarial safety still comes from K
independent sources and the SQIsign / STARK flavors.

The "whole chain in one proof" claim is flux-fold: an Ajtai/SIS lattice folding
scheme (transparent, post-quantum, NO trusted setup) that folds N block
commitments into ONE constant-size proof (~2,568 B regardless of chain length),
accepted in a single verify. Late-join catch-up is proven: a joiner verifies
one constant-size fold of the history prefix, adopts the checkpoint, and applies
only live blocks -- blocks_applied with divergence 0 over the skipped prefix.

== 2. A post-quantum keypair generated in a browser tab, no wasm ==

The wallet's hybrid identity (Ed25519 + Dilithium5 + Kyber1024) is now generated
client-side via @noble/post-quantum: ML-DSA-87 (= Dilithium5, FIPS 204) for
signatures, ML-KEM-1024 (= Kyber1024, FIPS 203) for key encapsulation. Pure JS,
audited, no WebAssembly module to fetch or fail. Measured: full hybrid keygen in
~267 ms in-browser, NIST Level 5.

The cautionary tale for anyone shipping PQ to browsers: we previously used the
`crystals-kyber` npm package, which crashes under rollup minification with
`ReferenceError: KeyGen512 is not defined` -- its internal functions are renamed
away by the bundler's scope hoisting. It works in dev and silently dies in a
production build. The fix was to drop it for a bundler-safe implementation. Flag
it if you maintain PQ JS libraries: survive minification, or you don't run in
production.

== 3. A committed-in-roots emission enforcer (structural anti-drift) ==

v0 described the four state roots as the structural fix for state divergence.
The same idea is now applied to monetary emission. SIGIL emission is a PURE
function of height (Bitcoin-style halving; 5 SIGIL/block, halve every 2,100,000
blocks, geometric sum approaching but never reaching a hard 21,000,000 cap).
There is no mutable "emission controller state" to drift -- which is the
postmortem fix for the class of bug where a production chain's PID-style
issuance controller once started at zero and emitted wrong rewards for ~3 days
before anyone noticed.

On top of the pure schedule sits a fail-loud band ENFORCER (not an actuator):
it compares realized native supply (read back from the committed wallet root)
against the deterministic target and HALTS if the drift exceeds +/-2% (200 bps).
A controller that mints is drift-prone; a verifier that halts turns "3 days of
wrong rewards" into "halt at block 1." The test `quillon_failure_is_caught`
encodes exactly that: zero-issuance against a large target -> immediate Err.
Schedule and cap are two independent guards (a compile-time const_assert that the
schedule sums to the cap, plus the runtime chokepoint that refuses any post-state
above 21M).

== 4. A verified-identity earning gate ==

New: sigil-rpc::onboard. A user binds an identity to their wallet by signing a
domain-separated, nonce'd message:

    sigil-onboard:v1: || wallet || '|' || method-tag || '|' || nonce

where method-tag is discord:<id>, oauth2:<provider>:<sub>, or self. The wallet
IS the public key, so the binding signature verifies directly. A VerifiedRegistry
admits only valid bindings (bad signatures rejected, not silently "verified").
The earning gate then routes mining/attestation rewards: a verified wallet earns
to itself; an unverified one routes to the dev-fee treasury. The reward is a
conserved pool->recipient transfer either way (the operator pool can never go
negative), and NATIVE is conserved in both branches (tested). The economic point:
a bound identity is the price of earning, and it costs the network nothing to let
unbound miners verify -- their reward just funds the treasury instead.

== 5. A two-of-N quorum with an LLM debate layer ==

flux-quorum now carries a `debate` module: an M-of-N cryptographic quorum with a
"proof-of-debate" layer on top, where two independent language-model agents must
agree before an agent-money action settles. The motivation is the same
eclipse-resistance arithmetic the light client uses -- P(bad) = f^K for K
independent sources -- applied to autonomous-agent decisions about money rather
than to peers. 11/11 tests. (This is research-grade; the cryptographic quorum is
the load-bearing part, the LLM debate is a policy layer above it.)

== Honest status / what's still open ==

  * Tip-proof SqiSignBlob + StarkRecursive flavors: still reserved wire tags;
    the running light clients use Blake3Fingerprint, which is integrity-only.
    Adversarial light-client safety is the SqiSignBlob/StarkRecursive lane.
  * "Full sync" in the running terminal client means lightweight succinct sync
    (verify the live tip + the fold proof + follow the stream to lag 0). It is
    NOT genesis-to-tip block download -- the testnet exposes a tip + a recent
    block stream, not a block-range endpoint. Called out so nobody over-reads it.
  * The flux-fold scheme's soundness rests on the Ajtai/SIS hardness assumption;
    independent review of the folding construction and its Fiat-Shamir transcript
    is welcome.
  * @noble/post-quantum carries its own audit; our usage is straightforward
    keygen/sign/verify, but the hybrid composition (Ed25519 + Dilithium5 +
    Kyber1024) glue is ours and worth a look.

Same invitation as v0: cryptanalysis on the tip-proof construction, the fold
scheme, the emission band enforcer's assumptions, and the PQ hybrid composition
