## Abstract

SIGIL is an experimental DagKnight chain built on the Flux compiler-as-platform. It is the sister network to Quillon Graph — *not* a replacement. Quillon stays on the production rails at `quillon.xyz`; SIGIL is the Flux-native laboratory.

SIGIL exists to prove three claims simultaneously:

1. **A blockchain whose binary is cryptographically provenance-signed at build time.** Every release artifact ships with an SQIsign-Level-5 signature over its content hash, source-tree hash, and producer wallet — generated by the compiler itself via `fluxc compile-native --provenance`. The chain commits to the producer's binary `.proof` in every block header.
2. **A blockchain where state divergence between nodes is impossible to hide.** Every block header commits four state roots (wallets, DEX pools, event log, contract storage) plus a STARK over the transition. A node that drifts on state diverges on roots and *halts itself* before serving traffic.
3. **A blockchain whose tip-verification fits in 10ms inside a browser.** Joining a SIGIL network does not require downloading the chain. A fresh node calls `flux-ivc-verifier-wasm` (or the equivalent native verifier) over a recursive STARK tip-proof, decides whether to trust the tip, and starts answering verified balance queries — without ever subscribing to `/sigil/g0/blocks`.

If those three hold for one real transaction routed across two nodes and a browser, SIGIL is a successful experiment.

This document describes the architecture, the foundation it stands on (Flux + content-hash compile cache + the FLUXFOOD app-build pattern), the prototype-by-prototype phased rollout that has shipped foundation work through Prototype 5 in a single 2026-05-29 session, and an honest account of what is currently real code versus stub-with-stable-wire-format.

## 2. Block header v0

The single most important schema in the protocol. See `SIGIL_GENESIS_v0.md` §2 for the full Rust struct. The interesting fields:

```rust
pub struct SigilBlockHeaderV0 {
    // identity
    pub version: u16, pub network_id: [u8; 8] /* b"sigil-g0" */,
    pub height: u64, pub parent_hash: [u8; 32], pub timestamp_ms: u64,

    // mining (SQIsign nonce-magic + Wesolowski VDF)
    pub nonce_sqisign: SqiSignature,      // SQIsign over (parent_hash || height || producer)
    pub vdf_input: [u8; 32], pub vdf_proof: WesolowskiProof,
    pub difficulty: u64,

    // state roots — THE Quillon fix
    pub wallet_state_root:   [u8; 32],
    pub dex_state_root:      [u8; 32],
    pub event_log_root:      [u8; 32],
    pub contract_state_root: [u8; 32],

    // transition correctness
    pub state_transition_proof: StarkProof,
    pub txs_merkle_root: [u8; 32],

    // provenance — the Flux dividend
    pub fluxc_artifact_proof: ProofBundle,

    // authorship, crypto-agile via flux-eternal-cypher
    pub sig_scheme: SigScheme,
    pub producer: ValidatorId, pub producer_sig: SignatureBytes,
}
```

**Three observations.** First, `nonce_sqisign` is *the* nonce: a SQIsign-Level-5 signature over `(parent_hash, height, producer)`. Two producers cannot collide accidentally (different pubkeys produce different sigs), an attacker who steals the VDF cannot replay a block (they don't have the producer's secret key), and the nonce doubles as an identity assertion — the block is provably authored by `producer` even before checking `producer_sig`. Second, the four state roots make divergence detectable at the protocol layer: a node computes locally, compares with the header, and halts on mismatch. Third, `fluxc_artifact_proof` binds the block to the *binary* that produced it — a `.proof` bundle of (artifact-BLAKE3, source-BLAKE3, agent-wallet, swarm-task_id, fluxc-version, fluxc-git, SQIsign-signature). This is the agility-attestation hook for the Flux build pipeline.

## 3. State agreement — the four-root primitive

Quillon nodes drifted on balances. The May 2026 incident dropped a wallet from 3200 → 1484 QUG and nobody noticed at the protocol layer because the header did not commit to "what the network agrees all balances are." SIGIL fixes this structurally.

| Root | Tree | Keyed by | Value |
|---|---|---|---|
| `wallet_state_root` | Sparse Merkle Tree | wallet address (32 B) | `balance: u128` |
| `dex_state_root` | Sparse Merkle Tree | `pool_id` (32 B) | `{reserves, lp_shares, accrued_fees, fee_bps, token_a, token_b}` |
| `event_log_root` | Binary Merkle, log₂N depth | event_index in block | `BLAKE3(event_encoded)` |
| `contract_state_root` | Sparse Merkle Tree | `(contract_id, slot_id)` | storage value (32 B) |

When a node receives block H it (a) verifies the state-transition STARK in ≤10ms, (b) applies transactions locally, (c) computes its own four roots, (d) compares with the header. Mismatch → log the diverging root + height, **halt the node** (refuse to serve API, refuse new txs), request canonical state from supermajority of peers, re-sync to last known good height.

The chokepoint (`sigil_state::commit_state_transition`) is the *only* function allowed to mutate `SigilState`. Every other write path is `pub(crate)`. The MIR-time gate (`flux_ai_audit`) enforces this at compile-release time; code that bypasses the chokepoint fails the audit and cannot reach a release binary.

### 3.1. Typed delta variants — P5-C (rocky-sigil-93)

The chokepoint accepts both raw `SetBalance`/`SetPool`/`PushEventHash`/`SetContractSlot` primitives and three typed delta variants for DEX operations:

- `StateMutation::SwapDelta { from, pool, in_token, in_amt, out_token, out_amt, fee, pool_after }`
- `StateMutation::LpDelta { from, pool, amt_a, amt_b, shares_minted, fee, pool_after }`
- `StateMutation::LpBurnDelta { from, pool, shares_burned, amt_a_out, amt_b_out, fee, pool_after }`

The chokepoint validates typed deltas as units before expanding to primitives: token pair matches the pool, immutable pool fields (token_a, token_b, fee_bps, lp_shares for Swap) stay byte-identical, reserves shift by exactly the declared amounts, sender's balance covers the operation with NATIVE-vs-non-NATIVE folding. Producers (sigil-tx → sigil-dex math) attach `pool_after` from their authoritative computation; the chokepoint catches the case where the delta got *munged* in transit. `CommitError::DeltaInvariant(String)` carries the specific violation. Five unit tests in `sigil-state/src/lib.rs` cover the happy path, reserve-shift mismatch, token-pair mismatch, first-deposit-creates-pool, and share-underflow rejection.

## 4. Privacy by default — sigil-mixer

Viktor's 2026-05-29 directive: every SIGIL transaction is shielded. There is *no opt-in path to transparent* at the protocol layer. Public ledger metadata is reduced to the minimum required for replay protection + fee payment + activation height.

### 4.1. Wire shape (`sigil-mixer/src/tx.rs`)

```
ShieldedSendTxData {
    input_nullifiers:   Vec<[u8; 32]>,    // owner-only, deterministic, no replay
    output_commitments: Vec<[u8; 32]>,    // BLAKE3 in P0, Pedersen on ark-bn254 in P1
    fee:                u128,
    token_hint:         TokenId,
    proof:              Vec<u8>,          // STARK from flux-zk-stark
}
```

The validator side checks: (1) every `input_nullifier` has not been spent (`ShieldedPool::spent_nullifiers` is a BTreeSet today, will migrate to flux-db CFs + SMT in P3), (2) every `output_commitment` opens to a valid value under the proof, (3) the proof verifies — which under the `real-zk` feature flag is real flux-zk-stark FRI + constraint + commitment verification, not a stub.

### 4.2. Phase 1 — real post-quantum STARK proving + verification (rocky-updater-92)

Behind `cargo --features real-zk` (default off so sigil-tx integration builds in 9.6s instead of 50s), sigil-mixer composes `flux-zk-stark::WalletPrivacyStarkProver` for end-to-end proving and verification. Five integration tests in `tests/stark_roundtrip.rs` cover the happy path, insufficient balance (prover errors before generating), all-garbage proof bytes (verifier rejects via FRI), empty proof bytes (verifier rejects), and determinism (different amounts → different tx_commitments and nullifiers). Plus the 26 unit tests from Phase 0, sigil-mixer carries **31 tests**.

The Phase 1 choice of `flux-zk-stark` over `flux-zk-snark`: the wallet_privacy layer in `flux-zk-snark` returns `Ok(true)` for any 128-byte blob — stubbed at the wallet-privacy verifier. `flux-zk-stark::wallet_privacy_stark::verify_transaction_privacy_stark` is real: full FRI + constraint + commitment via `StarkVerifier::verify`. Composing the real layer was the right Phase 1 call.

### 4.3. Known limitations (called out for the skeptic)

- Phase 0 commitments are **BLAKE3 over (value, token, blinding)**, not Pedersen on a real curve. Wire format stays stable when Pedersen lands in P1.
- Phase 1 supports **single-input single-output** ShieldedSend (one wallet, one recipient, plus implicit change). Multi-input / multi-output needs a Merkle-aggregation circuit `flux-zk-stark` doesn't expose yet — Phase 2.
- Tamper detection is **coarse**: a single-byte change to `proof.stark_proof` can land in padding the verifier doesn't read; only all-bytes-garbage reliably trips FRI. Tighter detection is an upstream flux-zk-stark task.
- The shielded surface for DEX (`ShieldedSwapTxData`, `ShieldedLpDepositTxData`, `ShieldedLpWithdrawTxData`) is queued, not shipped — rocky-updater offered to extend sigil-mixer once P5-B/C settled (both have).

## 5. Light client — tip-proof + verify-before-sync

### 5.1. Wire shape (`sigil-tip-proof`)

```rust
pub struct TipProof {
    pub version: u16,                   // == 0 in v0
    pub network_id: Vec<u8>,            // == b"sigil-g0"
    pub height: u64,
    pub roots: StateRoots,              // the four committed roots
    pub flavor: TipProofFlavor,         // Blake3Fingerprint | SqiSignBlob | StarkRecursive
    pub signature: Vec<u8>,             // 32B fingerprint | 292B SQIsign | tip_proof_v2 bytes
}
```

Flavor table — wire shape is stable across all three, only the interpretation of `signature` changes:

| Flavor | Backing crypto | Status |
|---|---|---|
| `Blake3Fingerprint` | BLAKE3 over canonical signing-bytes | v0 ✓ (rocky's sigil-tip-proof crate, 11/11 tests) |
| `SqiSignBlob` | SQIsign5 producer signature over signing-bytes | P4.1 (reserved tag) |
| `StarkRecursive` | `flux-recursive-proofs::tip_proof_v2` over chain prefix | P4.2 (reserved tag) |

The `Blake3Fingerprint` flavor is **integrity-only**: it catches wire bit-rot and non-canonical encodings, but does *not* prove the producer is honest. Joining nodes that need adversarial safety wait for `SqiSignBlob` or `StarkRecursive` before exposing a `/balance` HTTP API. `TipProofFlavor::adversary_resistant()` surfaces this in code; v0 returns `false` for the BLAKE3 flavor.

### 5.2. Producer-side emission — P4-A (rocky-sigil-84/85)

`sigil-node produce-block <txs.json> --broadcast` runs the existing broadcast cycle (spin up flux-p2p NetworkManager, 5s peer discovery, publish on `/sigil/g0/blocks`, 3s gossipsub propagation, shut down) plus a parallel `TipProof::new_blake3` publish on `/sigil/g0/tip-proofs` in the same network cycle. Receivers subscribed only to `/tip-proofs` (the P4-B `sigil-node join` flow, rocky's claim rocky-83) decide trust without ever seeing the block bytes.

A pre-flight `--broadcast --dry-run` flag runs the wire-format catcher: serializes both block and tip-proof, parses them back, compares hashes, fails loudly on mismatch — catches silent encoding bugs (the u128 / arbitrary_precision class) *before* committing to a 10s peer discovery + propagation cycle.

### 5.3. Browser-side verification — P4-E (rocky-sigil-88)

A single static HTML file at `sigil/gui/dist/verify-tip.html` (14.8 KB, no build step) implements the verifier for `Blake3Fingerprint`. It mirrors `sigil_tip_proof::TipProof::signing_bytes` byte-for-byte:

```
version_le · len(network_id)_le · network_id · height_le · 4×32B roots
```

The page imports BLAKE3 via `@noble/hashes` from `esm.sh`, lets the user build a proof + tamper its height or wallet_state_root, times the verify with `performance.now()` (sub-millisecond for the BLAKE3 flavor), and renders the result in obsidian + violet (per genesis §12a) with a violet-to-gold pulse on success. The forward-compat path to `flux-ivc-verifier-wasm` for the `StarkRecursive` flavor in P4.2 is documented inline — the JSON wire shape doesn't change; the JS dispatch on `proof.flavor` gains one arm.

The page is served by `fluxc serve` with `FLUX_STATIC_DIR=sigil/gui/dist` — no Python, no Node, no Caddy. `fluxc serve` is the only allowed dev server in the Flux ecosystem (rocky-sigil-89). The compiler IS the web server.

### 6.1. flux-db (sigil's storage)

`flux-db` is the Rust-native LSM-tree database SIGIL uses instead of RocksDB. The 2026-05-29 audit (`FLUX_DB_AUDIT_v0.md`, rocky-sigil-59) inventoried what's covered vs gaps. Two Phase-2 blockers shipped in the same session:

- **G1 — multi-CF atomic WriteBatch** (rocky-sigil-61). 70/70 flux-db tests pass including the in-process atomicity guarantee proven by `test_writebatch_concurrent_reader_sees_all_or_none` and the deadlock-free property under reverse-order concurrent batches (path-sorted lock ordering, proven by `test_writebatch_path_sorted_lock_order_no_deadlock`).
- **G2 — reverse iteration** (rocky-sigil-68). `DbIterator` is now a `DoubleEndedIterator`; `Database::iter_rev()` and `iter_from_back(end)` expose the rocksdb `IteratorMode::From(end, Reverse)` shape. 77/77 tests pass including the headline turbo-sync tip-lookup pattern (8-byte BE heights, find largest height ≤ N — reorg-rollback shape).

SIGIL Phase 2's state-transition chokepoint can now atomically multi-CF write; DagKnight round-descending vertex enumeration is unblocked; reorg-rollback walks work. Mainnet-class gaps (G4 BackupEngine, G1.5 cross-WAL crash atomicity) remain documented + queued.

### 6.2. fluxc compile cache — content-hash, cross-workspace

`fluxc-core::wrapper_mode` was marketing without code until 2026-05-29. The wrapper was a pure rustc passthrough; the log line `"RUSTC_WRAPPER=self active (flux-driver caching)"` was untrue. Three patches changed that:

- **Patch 1 — wiring** (rocky-sigil-75). `wrapper_mode` now calls `flux_cache::compute_hash` → `lookup` → `flux_driver::apply_cached_outputs` → fall through to rustc → `collect_outputs` → `store`.
- **Patch 3 — byte-level caching** (rocky-sigil-78). `flux-driver::collect_outputs` scans `--out-dir` for the real rustc-emitted file names (`lib<crate>-<hash>.rmeta` etc.) instead of guessing wrong names, then side-blobs the bytes to `target/flux-cache/blobs/<source_hash>/`. `apply_cached_outputs` reads them back and returns a real `bool` so the wrapper either exits 0 (cache HIT, skip rustc entirely) or runs rustc for real.
- **Patch 4 — cross-workspace key normalization** (rocky-sigil-82). Substitutes absolute paths in `--extern name=PATH` and `-L kind=PATH` with the BLAKE3 of the referenced rlib (or sorted file-name listing for directories); drops `--out-dir` from the key entirely (doesn't affect what gets compiled). Now byte-identical deps at different absolute paths produce identical cache keys. sigil/ rebuilds can hit cache entries populated by flux/ self-builds.

**Measured impact** at the moment of writing: cache size 7.17 GB of warm artifacts. The same `fluxc build --package flux-cache` that took 19.25s with cold cache returns in 3.10s on a re-run — 6× speedup, verified, intra-workspace. Cross-workspace gain unlocks the moment sigil/ hits a cache entry deepseek (or any other workspace) populated.

### 6.3. flux-p2p gossipsub mesh (rocky-sigil-86)

The P3 demo blocker — `drain_events` surfaced zero `GossipsubMessage` events even though producer `publish()` succeeded — was diagnosed in this session as libp2p's default mesh thresholds `mesh_n_low=4` + `mesh_outbound_min=2` deadlocking 2-3 node networks. ### 6.5. Master-wallet protocol fee plumbing — sigil-bank (rocky-94)

Per Viktor's directive, the master-wallet fee policy is baked into the protocol from day one, not bolted on as a governance lift later. The constants live in `sigil-bank`:

- `MASTER_MINING_FEE_BPS = 500` — 5% of mining rewards to master
- `MASTER_SWAP_FEE_BPS = 5` — 0.05% (5 basis points) protocol skim on swap output

`sigil-state::SigilState::master_wallet: Option<WalletId>` is settable exactly once via `StateMutation::SetMasterWallet { wallet }`. ## 7. Scoring — three-axis composite

`sigil-scoring` (rocky-sigil-64) brings Flux's SAP + X-Algo scoring rails into SIGIL keyed on Validator / Pool / Wallet / LP position subjects, plus a third axis ported from q-narwhalknight `void-walker::k_parameter` with chain semantics:

- **SAP** — 5 dims (contribution / latency / stake / accuracy / uptime), ported from `flux-p2p::sap`
- **X-Algo** — 5 cross-dims (temporal trust / consensus alignment / tx quality / topology rank / economic efficiency)
- **K-parameter** — physics-engine correlation series driven by network-activity amplitude (tx-rate) + SigilEvent kind tag instead of EEG + free-text intent; preserves drift_rate + oscillation_freq + phase_coherence + Gaussian noise + sliding history

Default composite: `0.5 · SAP + 0.3 · X-Algo + 0.2 · K-correlation`, tunable. For DEX swaps the three axes are populated as:

- **SAP axis** ← average of `pool_health.total` and `counterparty_trust = 0.6 · sender_sap + 0.4 · sender_xalgo`
- **X-Algo axis** ← `effective_price = 1 - slippage_bps/10000` (execution quality is intrinsically cross-algorithmic)
- **K axis** ← `kparam.normalised_correlation()` snapshot at swap time

The score is intended to be committed to the event-log root inside every `SwapExecuted` event — so swap quality becomes verifiable via Merkle inclusion against the tip-proof-attested root, not a number the wallet UI advertises in good faith. 25 tests across 4 modules, 100% clean dependency boundaries by the architect-score audit (rocky-sigil-flux foundation diagnostics).

## 9. The DEX — sigil-dex (P5-A, rocky-87)

Constant-product AMM math at `sigil-dex`: `swap(pool, direction, amount_in, min_amount_out)`, `add_liquidity(pool, amt_a, amt_b, fee_bps)`, `remove_liquidity(pool, shares)`. u128 reserves, fee in basis points, all arithmetic `checked_*`. `DexError` covers `ZeroAmount`, `EmptyPool`, `MathOverflow`, `SlippageExceeded`, `ReserveFloorViolated`, `KInvariantViolated`, `FeeTooHigh`, `InsufficientShares`, `RatioMismatch`. 23/23 tests pass. No I/O, no async — pure functions; `sigil-tx` and `sigil-state` are the only callers. The Reserve Floor (defaults around 1000 base units) is the deliberate guard against shallow-pool drainage on a single swap; production pools are expected to bootstrap with deep reserves.

`sigil-tx::Swap`, `LpDeposit`, `LpWithdraw` route through sigil-dex math (P5-B). Direction is picked from the pool's `(token_a, token_b)` pair against the tx's `in_token`; mismatch returns `TxApplyError::InvalidSwapToken` loudly, never silently swaps "the other side." First LpDeposit defines `(token_a, token_b, fee_bps)`; subsequent deposits verify against the existing pool and reject `PoolMismatch` before touching balance checks (so a user gets the right error message, not a wrong-token InsufficientBalance).

## 13. What would convince a skeptic

This document closes with the criteria for the experiment to be considered successful, as a contract between the operator and the agents.

1. **One real swap settles end-to-end on `sigil-g0`** — Alice's wallet sends 100 SIGIL to Bob via a swap routed through a SIGIL pool, the chain commits the SwapExecuted event into `event_log_root`, two nodes accept the block, and Bob's wallet shows the verified output.
2. **A fresh third node joins via tip-proof and never sees `/sigil/g0/blocks`** — boots cold, subscribes to `/sigil/g0/tip-proofs`, decides trust, answers a `GET /api/v1/balance?wallet=alice` with a Merkle inclusion proof against the tip-proof-attested wallet_state_root, and the joining node's verify-time is ≤10ms on commodity hardware.
3. **A divergence demo halts** — one node is patched with a tamper variant that injects `+42 SIGIL` to a demo wallet on block-receive. Block 1 arrives; the tampered node's wallet_state_root no longer matches the header. The node prints `🔴 STATE DIVERGENCE` and refuses to accept further blocks, exiting cleanly with code 78 (Q_PREFLIGHT_FAIL). The operator's alarm fires because the systemd unit doesn't auto-restart on 78.
4. **A release upgrades all running nodes via the gossipsub auto-updater** — `sigil-updater publish --binary sigil-node-v0.0.5 --proof sigil-node-v0.0.5.proof --version 0.0.5 --topic /sigil/g0/release`, the running nodes verify the SQIsign signature, download, verify BLAKE3, and `exec` themselves at the activation height. Block production never stops.
5. **A user's swap on a public SIGIL DEX has its quality badge verifiable in the browser** — open `https://sigilgraph.com/verify-swap.html?tx=…`, the page loads `flux-ivc-verifier-wasm`, fetches the event-log inclusion proof, checks the swap quality score against the committed root, shows ✓ or ✗ in violet-on-obsidian within one frame.

If five hold simultaneously, SIGIL has done what it set out to do.
