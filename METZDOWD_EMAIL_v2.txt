Subject: SigilGraph status update -- full verifiable genesis->tip sync (~7k headers/s, post-quantum roots), and a memory-bounded chain that scales in constant RAM

Follow-up to the v1 update (2026-06-01) and the v0 announcement (2026-05-29).
Two things changed materially since v1, and both CLOSE caveats I flagged last
time rather than open new claims. The five running items from v1 still hold;
they are condensed at the end so this reads standalone. Everything below is
reproducible against the live testnet (network id sigil-g0, ~975,000 blocks and
climbing) with curl and a browser.

The short version: last time the running light client did succinct tip-only
verification and I explicitly said it was NOT genesis-to-tip download. That
caveat is now gone -- there is a dedicated backfill channel and a light client
that verifiably reconstructs the whole chain at ~7,000 headers/second -- and the
producer that serves it now runs in bounded memory regardless of chain length.


== 1. The "no genesis->tip download" caveat is CLOSED: verifiable sync at ~7,000 blocks/s ==

v1's honest-status section said, verbatim, that "full sync" in the running
terminal client meant lightweight succinct sync (verify the live tip + the fold
proof + follow the stream), NOT genesis-to-tip block download, because the
testnet exposed only a tip and a recent stream. There is now a real backfill
channel and the client reconstructs and checks the entire chain.

The cryptography- and systems-relevant design:

  - Transport is a libp2p request-response channel (point-to-point), NOT gossip.
    This is a deliberate correction of an earlier gossip-re-broadcast backfill
    that amplified a server's serialize load by a factor of (peers+1) and
    collapsed block production under catch-up. A deterministic timing model made
    the failure legible before it could be re-shipped: serve-load = rate*(N+1)
    for the gossip path versus rate*1 for request-response, and only the latter
    stays under the producer's processing budget. Point-to-point + a server-side
    rate limit on the expensive (full-block) serves keeps production alive while
    serving catch-up.

  - A light client only needs HEADERS. Each SigilGraph block header carries the
    four committed state roots {wallet_state_root, dex_state_root,
    event_log_root, contract_state_root} plus the fields the tip-proof
    fingerprint binds (version, network id, height, the four roots). So the
    client requests headers_only and the server replies with bincode-encoded
    headers -- roughly 20x less wire than full-block JSON, and no JSON to lex.
    Full blocks (state transitions, the STARK attestation) are only requested by
    validating nodes (headers_only = false). Same wire, two consumers.

  - The client verifies the canonical tip-proof fingerprint (the BLAKE3 encoding
    from v1, item 5 below) over each received header's roots, and tracks a
    CONTIGUOUS synced height that is committed to disk (a height index plus a
    persisted pointer). A restart or an in-place upgrade therefore RESUMES from
    the verified prefix instead of re-downloading -- the pointer advances only
    over a contiguous run of present heights, so out-of-order parallel chunks
    fill gaps without ever reporting a non-contiguous height as "synced." This
    fixed a real bug where the client resumed from "highest height seen" (which
    a single stray gossip block inflates) and re-walked from zero.

  - Routing converges to whoever is actually serving: a peer that cannot serve a
    requested range (e.g. a node still catching up itself) is benched after two
    consecutive misses, with a short per-request timeout, so one slow or dead
    peer cannot gate a fan-out cycle. With a single healthy server the client
    still sustains thousands of headers/second.

Measured on the live 4-peer mesh: genesis-forward header sync at ~7,000
blocks/second (73,731 verified headers in ~10.6 s, in one run). The local
ingest path -- store the header, advance the committed contiguous pointer --
runs at ~26,000 blocks/second, so the bound is transport, not verification.

Honest scope, unchanged from v1 and stated plainly: headers-only sync verifies
INTEGRITY and network-binding (the BLAKE3 fingerprint over the committed roots
and the contiguity of the verified prefix). It does NOT by itself verify the
per-block state-transition (that needs the full block and the STARK flavor) nor
a producer signature (Phase 0 omits signature verification). Adversarial
light-client safety remains the reserved SqiSignBlob / StarkRecursive lane plus
K-independent-source arithmetic, exactly as in v1. What is new is that the
succinct, integrity-checked prefix now spans genesis->tip, fast, and resumably.


== 2. A memory-bounded chain: the producer scales to an unbounded chain in constant RAM ==

The producer that serves the above used to hold the ENTIRE chain in RAM (a
vector of every block) and snapshot/replay all of it; as the chain grew it was
OOM-killed in a restart loop (observed: 1,075 consecutive restarts). That is now
fixed structurally, and the fix is what makes serving historical headers cheap.

  - Persistence is an append-only on-disk block log: each block is one
    length-prefixed record, appended in O(1), with a small in-RAM offset index
    (one u64 per block). No periodic whole-chain re-serialization (that earlier
    O(N)-per-snapshot, fired every N blocks, was an O(N^2) production drag and is
    gone).

  - The in-RAM chain keeps only a bounded sliding WINDOW of recent blocks; older
    blocks live on disk and are read back by a single sequential range read when
    served. State (the four roots' backing structures) stays resident; raw
    history does not.

  - Recovery STREAMS the log one block at a time to rebuild state -- bounded RAM
    regardless of chain length. Proven live: working-set RAM stayed flat (tens of
    MB) while the on-disk log grew past a quarter-gigabyte; a restart
    stream-recovered ~102,000 blocks with the in-RAM window holding exactly the
    last 8,192 (window base == tip - 8192) and no OOM.

The recovery is verify-don't-trust in the same spirit as the rest of the system:
the log is the block stream, each record re-deserializes to a typed block, and
state is re-derived (not trusted from a snapshot blob). It is a durability/
systems result rather than a cryptographic one, but it is the reason a node can
cheaply serve the genesis->tip headers in item 1 without paging the whole chain
into memory.

A testing note that may interest anyone building verifiable systems: our
deterministic simulator ("chronos") covered the NODE's consensus path (apply,
divergence, catch-up timing) and caught real bugs there -- but it did NOT cover
the light-client's own sync/store, which is a separate binary, so those bugs
surfaced only in slow live runs. The lesson we wrote down: a green
consensus-layer simulation is not a correct client; each layer that syncs needs
its own deterministic test. The first such test (contiguous-pointer advance /
gap-stop / resume) immediately paid for itself by proving the store correct and
redirecting the hunt to the transport.


== 3. The v1 items, still running (condensed) ==

  (a) Light client / tip-proof. A client recomputes, byte-for-byte, the
      canonical fingerprint
        fp = BLAKE3( version(u16 LE) || len(network_id)(u32 LE) || network_id
                     || height(u64 LE) || wallet_root || dex_root
                     || event_log_root || contract_root )   [each root 32 B]
      in ~3 microseconds with 0 blocks downloaded. A browser implementation
      (vendored @noble/hashes BLAKE3) computes the identical value. The
      "whole chain in one proof" is flux-fold: an Ajtai/SIS lattice folding
      scheme (transparent, post-quantum, no trusted setup) folding N block
      commitments into one ~2,568-byte proof; late-join verifies one fold of the
      history prefix, adopts the checkpoint, applies live blocks, divergence 0.

  (b) Post-quantum keypair in a browser tab, no wasm. Hybrid identity
      (Ed25519 + ML-DSA-87 / Dilithium5, FIPS 204 + ML-KEM-1024 / Kyber1024,
      FIPS 203) via @noble/post-quantum, full hybrid keygen ~267 ms, NIST L5.
      Cautionary tale repeated for the list: the `crystals-kyber` npm package
      dies under bundler minification (ReferenceError: KeyGen512 is not defined)
      -- works in dev, silently fails in production. PQ-JS maintainers: survive
      scope-hoisting or you don't run in prod.

  (c) Committed-in-roots emission. Emission is a pure function of height
      (5 SIGIL/block, halve every 2,100,000 blocks, geometric sum -> hard 21M
      cap), so there is no mutable issuance controller state to drift. A fail-
      loud band ENFORCER (not an actuator) compares realized native supply (read
      from the committed wallet root) to the deterministic target and HALTS at
      +/-2% drift. Schedule and cap are independent guards (a compile-time
      const_assert the schedule sums to the cap, plus a runtime chokepoint that
      refuses any post-state above 21M). This is the postmortem fix for a real
      production chain whose PID-style issuance once started at zero and emitted
      wrong rewards for ~3 days: a verifier that halts turns that into "halt at
      block 1."

  (d) Verified-identity earning gate. A wallet binds an identity by signing a
      domain-separated, nonce'd message
        sigil-onboard:v1: || wallet || '|' || method-tag || '|' || nonce
      (method-tag in {discord:<id>, oauth2:<provider>:<sub>, self}); the wallet
      IS the public key, so the binding verifies directly. Verified wallets earn
      to themselves; unverified miners' rewards route to the dev-fee treasury.
      NATIVE is conserved in both branches (a pool->recipient transfer that can
      never go negative), tested.

  (e) Two-of-N quorum with an LLM-debate policy layer. An M-of-N cryptographic
      quorum carrying a "proof-of-debate" layer where two independent
      language-model agents must agree before an agent-money action settles --
      the same P(bad)=f^K eclipse arithmetic the light client uses, applied to
      autonomous-agent money decisions. The cryptographic quorum is load-
      bearing; the LLM layer is policy above it. Research-grade.


== Honest status / what is still open ==

  * CLOSED since v1: genesis->tip verifiable sync (item 1). The succinct,
    integrity-checked prefix now spans the whole chain, fast and resumable.
  * Still reserved: the tip-proof SqiSignBlob and StarkRecursive flavors (wire
    tags only). The running clients use the BLAKE3 fingerprint, which is
    integrity + network-binding, NOT adversary-resistant on its own (any peer
    can fabricate one). Adversarial safety = K independent sources + the
    SQIsign/STARK flavors. Headers-only sync inherits exactly this scope.
  * Phase 0 still omits per-block crypto verification (producer signature, VDF,
    STARK transition-validity) on the consensus path; those are the P1+ lanes.
  * flux-fold soundness rests on the Ajtai/SIS hardness assumption; independent
    review of the folding construction and its Fiat-Shamir transcript is welcome.
  * @noble/post-quantum carries its own audit; our usage is straightforward
    keygen/sign/verify, but the Ed25519 + Dilithium5 + Kyber1024 hybrid glue is
    ours and worth a look.

Same invitation as before: cryptanalysis on the tip-proof construction, the fold
scheme, the emission band enforcer's assumptions, and the PQ hybrid composition
is especially welcome. Reproduce it: the testnet feed is a public curl
(sigilgraph.fluxapp.xyz/sigil-status.json), the light client / terminal node is

content-addressed by BLAKE3 in sigil-top-latest.json), and the browser node
