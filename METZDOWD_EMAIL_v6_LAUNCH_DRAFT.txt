== 0. The crypto, up front (everything below is reproducible; URLs in section 5) ==

  - Hybrid provenance signatures, defense-in-depth: a require-BOTH composite of
    SQIsign (post-quantum isogeny, sqisign-rs 0.3, NIST L5, 292-byte sig /
    129-byte pk) AND Ed25519 (classical, 64-byte sig / 32-byte pk) -- a ~356-byte
    signature, STILL smaller than ONE ML-DSA-87 (Dilithium-5) signature at
    4595 B. The verifier requires EVERY leg in a fixed required scheme-set to
    validate (no single-leg downgrade, no "fall back to whichever still
    verifies"), so a break in either family -- an isogeny cryptanalysis advance,
    or a quantum attack on the curve -- does not forge a binary. The reasoning
    for hedging isogeny specifically: it is the youngest PQ family, sqisign-rs is
    0.x, and a sibling (SIDH/SIKE) fell completely in 2022; Ed25519 is the cheap
    classical belt-and-suspenders against an isogeny break or an implementation
    bug. The compiler's provenance path produces the bundle when the agent key is
    configured; no separate CI signer. This is OUT-OF-BAND release provenance,
    verified once at update time -- it is NOT in the per-tx batch-auth or
    turbo-sync path and cannot regress throughput or sync. [VERIFY @ v1.0:
    release artifacts carry non-empty SQIsign AND Ed25519 fields; the verifier
    enforces the required scheme-set (a single-leg bundle is REJECTED, not
    downgraded); the proof hash is committed in the block header.]
    (A third lattice leg, ML-DSA-87, slots into the same container for root keys
    that want isogeny-AND-lattice diversity, at the cost of size.)

  - flux-sigil: a SQIsign-keyed BLAKE3 -- a streaming hasher that auto-signs
    its own digest with the L5 key on finalize. Hash and authenticate in one
    pass.

  - Four sparse-Merkle state roots per block (wallet, dex, event log, contract
    storage). A node that drifts on any root diverges on the header and HALTS
    itself (exit 78) before it serves a byte. State divergence cannot hide.

  - flux-fold: a transparent, post-quantum Ajtai/SIS lattice folding scheme
    (NO trusted setup) that folds N block commitments into one compact folded
    proof object, verified in a single check. Measured on the zk-flux v0.3 lane:
    N=100k blocks -> ~2,568-byte proof, ~342 ms verify (representative point,
    not an asymptotic guarantee). Late join = verify one fold of the history
    prefix, adopt the checkpoint, apply live -- divergence 0.

  - A browser tip-check in two layers: the shipped light path recomputes the
    exact BLAKE3 tip fingerprint a native node commits, with 0 blocks
    downloaded; the PQ proof path is the flux-ivc-verifier-wasm verifier
    (STARK + lattice) with a 10 ms gate. [VERIFY @ v1.0: which path is enabled
    by default in the browser release.] In-tab hybrid PQ identity -- Ed25519 +
    ML-DSA-87 + ML-KEM-1024 (FIPS 204/203; the PQ members sit at NIST L5),
    full keygen ~267 ms in-browser, no wasm.

  - chronos: a deterministic, seeded, virtual-time harness that runs the REAL
    chain code and measures the CPU-time cost of its real crypto paths. It
    times actual ed25519 batch-auth (keygen + sign + verify, one signature per
    N-op batch, parallel across cores) head-to-head against a zk-flux
    verify-once path. The zk-flux numbers are real prove/verify times and real
    proof/wrap byte sizes, but a COST STRUCTURE only: the lattice wrap binds the
    FRI-STARK commitment, it does not yet PROVE STARK validity. A 72-hour soak
    resolves in seconds of wall clock and is bit-reproducible.

  - Time-based emission: block_reward = integral of rate over
    [prev_block_ts, block_ts]; rate halves every 4 years; geometric sum to a
    hard 21,000,000 cap. Invariant under block rate -- a height-based schedule
    breaks the instant block cadence changes (we found this at 220 blk/s).

== 1. The substrate: a compiler that signs what it builds ==

Flux is a self-hosting Rust build orchestrator (content-hash compile cache,
measured 6x intra-workspace cache speedup; Kahn's-algorithm parallel batching;
a MIR -> CLIF native-object backend for a growing subset of Rust, demonstrated
end to end). The primitive relevant to this list: a release binary can carry a
`.proof` bundle whose signing input is a canonical 177-byte record:

    version || artifact_blake3 || source_blake3 || fluxc_version ||
    fluxc_git || agent_wallet || swarm_task_id || settle_tx || timestamp_us

The record is signed with SQIsign Level 5 (292-byte signature, 129-byte public
key) when the compiler has the agent key; verification first recomputes the
artifact BLAKE3, then checks the SQIsign signature over the canonical record.
The same crate rejects downgraded-length signatures before parsing (for
example, a 148-byte Level-1 signature is a hard error, not "false"). The
flux-sigil helper has the analogous content format for data streams:
`32-byte BLAKE3 digest || 292-byte SQIsign-L5 signature over that digest`.
This is the wire-format claim, not a slogan.

Releases are content-addressed by BLAKE3 and the updater is fail-closed: a
client verifies the manifest hash end-to-end before swapping binaries, probes
the new binary's version BEFORE handing off, and auto-rolls-back if a freshly
updated version fails to reach a healthy run three boots in a row. The update
loop has been exercised in production weekly. A require-both SQIsign-L5 +
Ed25519 hybrid `.proof` is now LIVE for the shipping 0.77.6 release (section 5):
it binds the binary's BLAKE3 to the signing agent and verifies require-both.
That current proof was produced post-build -- it binds the artifact and the
agent, not the source -- because the matching source tree was not cleanly
available; the build-time, compiler-emitted artifact+source binding described
above is wired and demonstrated, awaiting a release cut from a current tree.
[VERIFY @ v1.0: the shipping release is built and signed in one pass so the
artifact AND the source are bound, and the proof hash is committed in-header.]

== 2. The chain: state divergence cannot hide, and the money cannot drift ==

Every block header commits four state roots (wallets, DEX pools, event log,
contract storage). A node that drifts on state diverges on roots and halts
itself before serving traffic -- the structural fix for the replay-bug class
that silently corrupted a production balance on our sister chain in May 2026.

Emission, v1.0's headline change, is now a pure function of TIME, not height:

    rate(t)        = R0 >> floor((t - t_genesis) / HALVING_PERIOD)
    block_reward   = integral of rate over [prev_block_ts, block_ts]
    HALVING_PERIOD = 4 years; geometric sum -> hard 21,000,000 cap

The postmortem that forced this: our testnet producer runs verify-once empty
blocks at ~220 blocks/second for throughput research. Height-based halving
(Bitcoin's schedule) at that rate burns one full 2.1M-block halving epoch every
~2.6 hours -- the entire 21M schedule emitted in HOURS. The lesson generalizes:
a height-based schedule embeds an assumption about block cadence; break the
cadence and you break the money. Time-based emission makes the schedule
invariant under block rate. Sub-second blocks earn the integral of the rate
over their wall-clock slice, with the sub-unit remainder carried across blocks
so no emission is lost to integer truncation. [VERIFY @ v1.0: chronos gate
green -- cumulative emission curve IDENTICAL at 220 blk/s and 2 blk/s,
approaches 21M asymptotically, never exceeds the cap, halves each period.]

Three independent guards survive from the earlier design: a compile-time
const_assert that the schedule sums exactly to the cap; a runtime commit
chokepoint that refuses any post-state above 21M base units; and a fail-loud
band enforcer that compares realized supply (read back from the committed
wallet root) to the deterministic target and HALTS at +/-2% drift. A
controller that mints is drift-prone; a verifier that halts turns "three days
of wrong rewards" (a real incident on a predecessor chain) into "halt at
block 1."

== 3. One binary: node + miner + wallet, on a potato ==

sigil-top is a single static binary (Linux + Windows) that is a monitoring
node, a light client, a dual-lane miner, and a wallet. Things the list may
find interesting because each encodes a hard-won invariant:

  - Light monitor by default: verifies the live tip fingerprint
      fp = BLAKE3(version || len(net_id) || net_id || height
                  || wallet_root || dex_root || event_root || contract_root)
    in ~3 microseconds with 0 blocks downloaded; a browser implementation
    (vendored @noble/hashes BLAKE3) computes the identical value. Full
    genesis->tip header sync is opt-in and sustains 8,000+ headers/second
    against the public testnet on consumer hardware (reference machine in
    section 5). v0.8.0 live-mesh receipt: 73,731 headers downloaded/stored in
    ~10.6 s, and full restart stream-recovered ~102,000 blocks in bounded
    memory. [VERIFY @ v1.0: do not claim 1M-block catch-up until the shipping
    build has a named run at that scale.]

  - Whole chain in one proof: flux-fold, an Ajtai/SIS lattice folding scheme
    (transparent, post-quantum, no trusted setup) folding N block commitments
    into one compact folded proof object, verified in a single check.
    Measured point (zk-flux v0.3 lane, representative): N=100k blocks ->
    ~2,568 B proof, ~342 ms verify. Late join = verify one fold of the
    history prefix, adopt the checkpoint, apply live blocks, divergence 0.

  - Anti-phantom sync: a peer-claimed tip may only RAISE the sync target
    within a bounded lead over a signed oracle tip; store watermarks are keyed
    to the genesis hash, so a testnet reset self-heals in seconds instead of
    leaving clients chasing a ghost height from the previous chain.
    [VERIFY @ v1.0: the reset acceptance test -- stale client + fresh-genesis
    producer self-heals < 10 s with no manual intervention -- is in CI.]

  - Bounded-RAM durability: the block store opens in O(window), never O(chain).
    A write-ahead log that grows past a sanity bound is quarantined at open
    rather than replayed into RAM -- the postmortem fix for a Windows
    append-mode quirk (Rust's append(true) strips FILE_WRITE_DATA, so set_len
    truncation silently fails) that let a 1.6 GB poisoned WAL eat 3-4 GB of
    RAM at every start. Recovery streams the log one block at a time; RAM
    stays flat (tens of MB) regardless of chain length.

  - PQ identity in a browser tab, no wasm: hybrid Ed25519 + ML-DSA-87 +
    ML-KEM-1024 (FIPS 204/203) via @noble/post-quantum, full keygen ~267 ms;
    the ML-DSA-87 and ML-KEM-1024 members are NIST L5, composed with Ed25519
    for hybrid/transitional security (the composition's strength is not itself
    a NIST level). (Repeated warning for PQ-JS maintainers: survive bundler
    minification or you don't run in production.)

  - Verified-identity earning: a wallet binds an identity by signing a
    domain-separated nonce'd message; the wallet IS the public key, so the
    binding verifies directly. Verified wallets earn to themselves;
    unverified mining routes to the treasury. A fixed 1.2% of every reward
    routes to a commons pool governed by an M-of-N quorum -- the cryptographic
    quorum is load-bearing; an LLM-debate layer above it is policy, not
    consensus.

== 4. The testing culture that found the bugs ==

Three practices paid for themselves repeatedly and are now permanent:

  - Deterministic virtual time ("chronos"): the harness runs the REAL chain
    code -- real apply_tx + commit_state_transition + the four roots -- under a
    simulated clock, so a producer and follower exchange real blocks over
    simulated hours and the whole run resolves in test-wall-clock seconds,
    deterministically and seeded. Crucially it does NOT stop at network timing:
    it measures real signature-verification CPU cost. The current ed25519
    batch-auth path (one signature authorizes a batch of N single-author ops;
    every non-producing node re-verifies it, parallel across cores) is
    benchmarked head-to-head against a zk-flux verify-once path, where the
    producer emits one PQ128 lattice-wrapped FRI-STARK proof and every verifier
    checks one O(1) wrap independent of the tx count. The numbers reported are
    real prove() time, real O(1) verify() time, and real proof + wrap byte
    sizes. Design sweeps (WAL bounds, value sizes, lock-free read paths,
    emission curve at two block rates) run in virtual time first; only the
    winner touches hardware. Written down after a live 5 TB run brought the
    build host to load 158: a green consensus-layer simulation is NOT a correct
    client -- every layer that syncs needs its own deterministic test.

    [Honest framing for this list: chronos measures signature-verification
     COST and CORRECTNESS (sign->verify roundtrip, and root-divergence
    rejection), not signature security. The auth path it benchmarks is
     ed25519; SQIsign-L5 is the provenance/tip-proof path, separate from
     batch-auth. The zk-flux lattice wrap BINDS the FRI-STARK commitment but
     does not yet PROVE STARK validity (the FRI-verifier-in-circuit is the
     recursion frontier), so the zk-flux figures are a cost structure, not a
     soundness claim.]

  - Stargate wind tunnel (not a v1.0 live-throughput claim): after the O(1)
    accumulator-root and parallel-execution work, `parallel_exec.rs` measured
    1,901,939 tx/s at 4 threads and 5,173,981 tx/s peak at 16 threads on
    Epsilon (4M tx, 1M accounts, release build). The correctness anchor is
    asserted at every thread count: the parallel commutative-merge root equals
    the serial fold root. This is the receipt behind "about 2M TPS," but it is
    a state-transition wind-tunnel number, not public-testnet TPS. The
    remaining Stargate path is DagKnight/Narwhal: many producers emit into a
    BlockDAG, Narwhal disseminates batches, DagKnight orders hashes. That lane
    is a harness/integration target, not shipped v1.0 consensus.

  - Operator-eye dogfooding: a sentinel exercises the system exactly the way
    a human operator sees it -- launches the real client, checks that sync
    ADVANCES, that the published checkpoint is fresh against the live tip,
    that the update channel is consistent, that no competing miner races the
    operator's rig, that the wallet balance endpoint answers -- and files
    findings automatically. Several of the fixes above were first caught by
    a human; each such miss became a sentinel check the same day.

  - Provenance as a test artifact: because the compiler emits the proof bundle
    next to the artifact, "which build produced this block, from which source,
    by which agent" is a BLAKE3/SQIsign check, not a log-grep, once signed
    agent keys are present. Reproducing a reported number starts by verifying
    the .proof of the binary that produced it.

== Honest scope (kept short; the list will check anyway) ==

  - chronos benchmarks the ed25519 auth path and a zk-flux cost structure;
    it is not a cryptanalysis harness and does not test signature forgery.
    SQIsign-L5 secures binary provenance and one tip-proof flavor, NOT the
    per-tx batch-auth measured in section 4.
  - The require-both hybrid is now IMPLEMENTED and LIVE, not a target: the
    Ed25519 leg is wired, each leg signs a domain-separated digest binding the
    (scheme, public-key) set, and the verifier enforces a required scheme-set --
    a single-leg-stripped bundle is rejected, never downgraded (the old
    accept-any-passing-leg fallback was removed). A require-both proof over the
    shipping 0.77.6 binary is published and verifies (section 5). This work is
    additive to the provenance API; it does not alter the existing SQIsign
    sign/verify surface or the ed25519 batch-auth path, so the throughput and
    turbo-sync numbers stand unchanged.
  - Two honest limits on that live proof: (1) it binds the artifact and the
    signing agent, NOT the source -- it was signed over the shipped binary
    post-build, not emitted at compile time, because the matching source tree
    was not cleanly available; the build-time artifact+source binding is wired
    and demonstrated, pending a release cut from a current tree. (2) in-header
    commitment of the proof hash remains the v1.0 target.
  - flux-fold's 2,568 B / 342 ms is a representative measured point on the
    zk-flux v0.3 lane at N=100k blocks, not an asymptotic or worst-case bound;
    proof-size growth vs N and the verify cost at other N are not claimed here.
  - The Blake3Fingerprint tip-proof flavor is integrity + network-binding,
    not adversary-resistant alone; adversarial safety = K independent
    sources plus the SQIsign / STARK flavors. [VERIFY @ v1.0: which of the
    SqiSignBlob / StarkRecursive flavors ships enabled.]
  - zk-flux's lattice wrap binds but does not yet prove STARK validity; the
    verify-once figures are cost-structure, not soundness.
  - flux-fold soundness rests on Ajtai/SIS; independent review of the folding
    construction and its Fiat-Shamir transcript remains very welcome.
  - The ~2M / ~5.17M Stargate numbers are measured state-transition harness
    numbers, not production network throughput; they exclude live mempool,
    storage, p2p, and final DagKnight ordering. DagKnight/Narwhal remains a
    named integration lane until a v1.0 harness or production run is cited.
  - No 1M-block sync claim is made here. The cited sync receipts are 73,731
    headers in ~10.6 s, 8,000+ headers/s on the current path, and ~102,000
    blocks stream-recovered in bounded memory.
  - The @noble/post-quantum hybrid glue (Ed25519 + ML-DSA-87 + ML-KEM-1024
    composition) is ours and worth independent eyes; "NIST L5" describes the
    PQ members, not the composition.
  - The testnet runs two chains by design (a verify-once throughput chain and
    a dual-lane mined rewards chain); the throughput numbers and the mining
    numbers in this post come from the chain named in each claim.

Cryptanalysis on the tip-proof construction, the fold scheme, the time-based

composition is explicitly invited.
