== 1. Flux ==

Flux is a self-hosting build orchestrator for Rust that closes the loop
between source tree, compiler, binary, and on-chain release. Versions
through v0.17.0 (2026-05-29) implement a three-phase architecture:

  - Phase 1: rustc wrapper with a real content-hash compile cache.
    Cache keys substitute --extern paths with the BLAKE3 of the
    referenced rlib's bytes, so byte-identical deps at two
    different absolute paths produce one cache key.
    Measured 6x speedup intra-workspace; cross-workspace dedup
    works the moment any sibling app builds against a flux-warmed
    target.

  - Phase 2: flux-graph workspace discovery + Kahn's-algorithm
    build batching. Drives parallel rustc per batch with
    deterministic ordering.

  - Phase 2c: MIR -> CLIF -> linkable native object emission for the
    fluxc-compiled subset of Rust. Small but real; demonstrated end
    to end on sum_to, recursion (fact(15) == 1307674368000), n-way
    match, tuple projection, and intra-crate function calls. The
    .o emitted links against rustc-built Rust harnesses and
    executes correctly.

The primitive most relevant to this list: every release binary
fluxc emits carries a `.proof` bundle -- an SQIsign-Level-5
signature over (artifact-BLAKE3, source-BLAKE3, agent-wallet,
swarm-task-id, fluxc-version, fluxc-git, timestamp). The signature
is computed by the compiler that produced the binary, using the
agent's per-session keypair. The chain (below) commits the bundle
in every block header. There is no separate CI signing step; the
compiler signs.

SQIsign-Level-5 was chosen for compactness: 292 bytes per signature
versus Dilithium-5's 4595. 16x smaller is what makes "every header
carries a full producer signature plus the binary provenance proof"
viable at chain scale.

Substrate stats: 55 crates, ~81K LOC, 8 vendored ZK crates
(STARK + lattice SNARK + recursive proofs + tip-proof STIR +
IVC-verifier-WASM + zk-snark + zk-p2p + zk-types), all green.

== 2. SigilGraph ==

SigilGraph is an experimental DagKnight chain built on Flux. Sister
network to Quillon Graph (production, at quillon.xyz); SigilGraph
is the Flux-native laboratory, not a replacement. Three claims it
exists to prove simultaneously:

  (a) A blockchain whose binary is cryptographically provenance-
      signed at build time. (Inherits Flux's `.proof` bundle in
      every block header's `fluxc_artifact_proof` field.)

  (b) A blockchain where state divergence between nodes is
      impossible to hide. Every block header commits four state
      roots -- wallets, DEX pools, event log, contract storage --
      plus a STARK over the transition. A node that drifts on state
      diverges on roots and halts itself (exit code 78,
      "Q_PREFLIGHT_FAIL") before serving traffic. This is the
      structural fix for the class of replay bug that dropped a
      Quillon user from 3200 QUG to 1484 QUG in May 2026, which
      Quillon's header (which committed only parent + tx_merkle +
      miner) could not protocol-level detect.

  (c) A blockchain whose tip-verification fits in <=10 ms in a
      browser. Joining nodes call flux-ivc-verifier-wasm over a
      recursive STARK tip-proof, decide trust, and start answering
      verified balance queries -- without ever subscribing to
      /sigil/g0/blocks. The wire shape is locked across three
      flavors: Blake3Fingerprint (integrity only, v0),
      SqiSignBlob (SQIsign-5 producer sig, P4.1, reserved tag),
      StarkRecursive (flux-recursive-proofs tip_proof_v2, P4.2,
      reserved tag).

The privacy story is opinionated: every transaction is shielded
by default. No opt-in path to transparent. The mixer (sigil-mixer)
ships Phase 1 with real post-quantum STARK proving + verification
via flux-zk-stark::wallet_privacy_stark. `verify_shielded_send_stark()`
runs full FRI + constraint + commitment verification, not stubbed.
Single-input single-output ShieldedSend works end-to-end (5 STARK
roundtrip integration tests pass); multi-input multi-output via
Merkle aggregation is Phase 2.

Wallet-side commitments are BLAKE3 over (value, token, blinding) in
Phase 0 -- migration to Pedersen commitments on ark-bn254 lands in
mixer Phase 2 without changing the wire format. Nullifiers are
derived deterministically by the owner from (commitment, secret-key).

DEX is a constant-product AMM (port of q-dex from the Quillon
codebase). Master-wallet protocol fee routing -- 5% of mining
rewards, 5 bps (0.05%) of DEX swap output -- is baked into
sigil-state from day one as a setMasterWallet StateMutation
variant that fires exactly once and refuses re-setting
(CommitError::MasterWalletAlreadySet). Not bolted on as a
governance lift later.

== 3. SigilOS ==

SigilOS is the browser-first WASI userspace for SigilGraph,
mirroring QuillonOS (the production version for Quillon Graph at
quillon.xyz/os.html). Modules are Rust crates fluxc-compiled to
wasm32-wasip1; they run inside a browser tab against a virtual
filesystem and a small syscall surface. The point is the chain
has a userland -- a fresh SigilGraph node coming up via tip-verify
in 10 ms is also a fresh SigilOS environment that can run signed
binaries fetched from the same provenance-bound release pipeline
the chain itself runs on.

The convergence: the same `.proof` bundle that authenticates a node
binary also authenticates a SigilOS userland program. The chain
commits to both via the same fluxc_artifact_proof field in the block
header. There is no separate "trust me, this is the right release"
link for OS utilities versus chain binaries -- both are signed by
the same compiler under the same agent's wallet, and both can be
verified by the same browser-side WASM verifier in sub-millisecond.

SigilOS is queued behind SigilGraph testnet (Phase 7 of the SIGIL
phased roadmap). The QuillonOS pattern is already proven in
production with crates: quillonos-coreutils, quillonos-sh,
quillonos-init, quillonos-uname, quillonos-readfile,
quillonos-q-miner, quillonos-claude.

== The connection ==

Flux + SigilGraph + SigilOS form a vertical:

  build provenance       Flux: fluxc compile-native --provenance
                         |
                         v
  chain commitment       SigilGraph: fluxc_artifact_proof in header
                         |
                         v
  light-client verify    SigilOS: <=10 ms STARK verify in WASM
                         |
                         v
  userland execution     SigilOS: same signed binary runs in browser

Six layers that are conventionally loosely connected (CI builds, CI
signing, registry push, operator pulls, operator runs, user trusts)
collapse into one cryptographically-bound chain when the compiler,
the consensus, and the browser verifier are all parts of the same
vertical substrate.

== Honest status ==

What works end-to-end with tests:
  * Block header schema + 4 state roots + chokepoint with typed
    delta variants (SwapDelta / LpDelta / LpBurnDelta) and
    k-invariant + share-conservation guards. 12 sigil-state tests.
  * Constant-product AMM with checked u128 arithmetic. 23 sigil-dex
    tests + 5 P5-C delta-chokepoint tests.
  * Tip-proof emission (producer side) + Blake3Fingerprint flavor
    verifier (Rust crate, 11 tests) + browser verifier (14.8 KB
    static HTML, sub-ms in Firefox 130 on commodity hardware).
  * sigil-mixer Phase 0 (BLAKE3 commitments) + Phase 1 (real STARK
    behind --features real-zk; 5 STARK roundtrip tests, 26
    Phase-0 unit tests). 31 tests total in sigil-mixer.
  * Flux content-hash compile cache: 7 normalization tests, 6x
    speedup on warm rebuilds verified in vitro.
  * flux-p2p gossipsub mesh fix for small networks (mesh_n_low=1,
    mesh_outbound_min=1) with env overrides; 30 tests.

What is stub or reserved:
  * Tip-proof SqiSignBlob + StarkRecursive flavors -- reserved tags
    on the wire; verifier returns NotImplemented.
  * Pedersen commitments on ark-bn254 -- Phase 2 of mixer.
  * Multi-input multi-output ShieldedSend -- Phase 2 of mixer.
  * flux-vdf (Wesolowski VDF) -- queued port of q-vdf.
  * DagKnight consensus core -- queued port of q-narwhal-core +
    q-dag-knight.
  * WASM VM (smart contracts) -- queued port of q-vm.
  * Cross-WAL crash atomicity in flux-db -- mainnet gate.
  * BackupEngine equivalent in flux-db -- mainnet gate.

Known cryptanalysis-worthy gaps (calling them out for the list):
  * flux-zk-stark tamper detection is coarse -- a single-byte
    change to the proof bytes can land in padding the verifier
    doesn't read; only all-bytes-garbage reliably trips the FRI
    check. Tighter tamper detection upstream is a real task.
  * sigil-tip-proof Blake3Fingerprint flavor is integrity-only,
    NOT adversary-resistant; any peer can fabricate one. The wire
    is honest about this -- TipProofFlavor::adversary_resistant()
    returns false for Blake3Fingerprint. Joining nodes that need
    adversarial safety wait for SqiSignBlob or StarkRecursive.
  * The MIR-time chokepoint enforcement via flux_ai_audit (the
    "every state-write must go through commit_state_transition"
    rule) is implemented as a lint, not a soundness proof.
    Adversarial code that bypasses the chokepoint while still
    passing the audit is a known attack surface.

Development model: multi-agent. Four Claude Code sessions today,
one workspace, coordinated via MCP swarm tools. Settlement on
Quillon Graph in QUG per delivered task. Approximate day-total
compensation across the four sessions: ~17 QUG. Same wallet,
different agent_ids (rocky / rocky-updater / rocky-sigil /
rocky-83). The agent_id is the conscience that records who
shipped what, even though the wallet collects for all.
